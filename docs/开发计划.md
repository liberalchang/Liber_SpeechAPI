# Liber_SpeechAPI 开发计划（ASR + TTS）

## 1. 背景与目标
本项目目标是在本仓库内实现两类能力，并提供两种使用形态：

- **语音转文字（ASR）**：参考 `insanely-fast-whisper`（基于 HuggingFace Transformers 的 Whisper ASR pipeline，支持批处理、时间戳、可选 Flash Attention 2、可选说话人分离）。
- **文字转语音（TTS）**：参考 `chatterbox-tts`（Chatterbox / Chatterbox-Turbo / Chatterbox-Multilingual，支持可选参考音频的零样本音色克隆）。

交付物必须覆盖：

- **远程调用 API 接口**：提供可部署的 HTTP 服务，供其他机器/程序远程调用。
- **本地快速调用 CLI 命令**：提供低门槛、快速的命令行使用体验。
- **快速安装配置脚本**：使用 **uv** 管理 Python 依赖；整体采用**交互式配置**，在 Windows 上优先提供 `PowerShell` 脚本。

补充要求：

- **多平台部署**：部署环境需要支持 Windows / Linux（以及可选 macOS）。
- **API 鉴权**：远程 API 需要支持鉴权（建议 API Key / Bearer Token 方式）。

## 2. 范围与非目标
### 2.1 范围
- ASR：音频文件或 URL 输入，输出文本及时间戳（chunk/word）；可选 diarization（说话人分离）。
- TTS：文本输入，输出 **wav / mp4 / ogg(opus)**；可选参考音频（voice prompt）；可选 Turbo / 标准 / 多语（默认多语）。
- CLI：提供 `transcribe/translate/synthesize/serve/doctor/warmup` 等命令。
- API：FastAPI + OpenAPI；支持上传文件与 URL；提供健康检查与预热。
- 安装脚本：交互式选择能力组件、设备类型（CPU/CUDA）、默认模型等；写入统一配置文件。

### 2.2 非目标（阶段 1 不做）
- UI（如 Web 前端、Gradio）不作为必做交付。
- 复杂的队列系统/分布式推理（可作为后续扩展）。
- 大规模多租户鉴权/计费系统（如需可后续加）。

## 3. 关键约束（来自参考文档）
### 3.1 ASR（insanely-fast-whisper）
- 核心是 `transformers.pipeline("automatic-speech-recognition")`。
- 性能优化点：`torch_dtype=float16` + batching；可选 Flash Attention 2（`attn_implementation="flash_attention_2"`）。
- 可选 diarization：依赖 `pyannote-audio`，通常需要 HuggingFace Token。
- Windows 上可能存在 CUDA/Torch 安装兼容问题，需要 `doctor` 命令给出诊断与指引。

### 3.2 TTS（chatterbox-tts）
- 安装方式：`pip install chatterbox-tts`。
- 使用方式：`ChatterboxTTS/ChatterboxTurboTTS/ChatterboxMultilingualTTS.from_pretrained(device=...)` + `model.generate(...)`。
- device：`cuda`/`cpu`（以及可选 `mps`）。
- 音色克隆：通过 `audio_prompt_path` 可选启用。

## 4. 总体架构设计
### 4.1 设计原则
- **核心能力复用**：ASR/TTS 逻辑只实现一份，API 与 CLI 只是“薄适配层”。
- **配置统一**：API、CLI、脚本使用同一套配置项（环境变量或配置文件）。
- **可选特性按需加载**：如 diarization、flash-attn、Turbo/Multilingual 等，避免默认安装过重或失败。
- **可观测与可诊断**：`doctor` 输出清晰的依赖与设备状态，定位 Windows CUDA/Torch 问题。
- **结果可下载**：远程调用场景下，推理结果（转录文件/音频文件）需要由请求方下载到本地。

### 4.2 推荐目录结构（拟定）
> 注：最终目录会根据你现有仓库命名习惯微调。

- `src/liber_speech/`
  - `engines/`：核心推理封装（权威逻辑）
    - `asr_engine.py`：Whisper ASR 封装
    - `tts_engine.py`：Chatterbox TTS 封装
    - `device.py`：设备选择与检测
    - `model_cache.py`：模型缓存目录、预热、下载检查
    - `io_types.py`：内部请求/响应数据结构
    - `errors.py`：统一错误类型
  - `api/`：远程服务
    - `main.py`：FastAPI app
    - `routes_asr.py`：ASR 路由
    - `routes_tts.py`：TTS 路由
    - `settings.py`：配置读取
    - `dependencies.py`：引擎单例与懒加载
  - `cli/`：命令行
    - `app.py`：Typer 入口
    - `commands_asr.py` / `commands_tts.py`
- `scripts/`
  - `setup.ps1`：交互式安装与配置（Windows 优先）
  - `setup.sh`：可选
- `configs/`：示例配置模板
- `tests/`：冒烟测试（CPU 友好）与可选 GPU 测试
- `docs/`：文档（本文档所在）

### 4.3 配置约定（拟定）
建议支持两种方式，二选一或同时存在：

- **`.env`（适合 API/部署）**
  - `LIBER_DEVICE`：`cpu` / `cuda:0` / `cuda:1` ...
  - `LIBER_ASR_MODEL`：如 `openai/whisper-large-v3`
  - `LIBER_ASR_BATCH_SIZE`：如 `24`
  - `LIBER_ASR_FLASH`：`0/1`
  - `LIBER_ASR_TIMESTAMPS`：`chunk/word`
  - `LIBER_DIARIZATION`：`0/1`
  - `LIBER_HF_TOKEN`：可空
  - `LIBER_TTS_MODEL`：`turbo/standard/multilingual`（默认 `multilingual`）
  - `LIBER_TTS_FORMAT`：`wav/mp4/ogg_opus`（默认 `ogg_opus` 或按你的偏好）
  - `LIBER_HF_CACHE_DIR`：可选（统一缓存目录）
  - `LIBER_API_AUTH_MODE`：`none/api_key`（默认 `api_key`）
  - `LIBER_API_KEYS`：允许的 key 列表（以逗号分隔或指向文件路径）
  - `LIBER_RESULTS_DIR`：服务端保存结果文件的目录
  - `LIBER_RESULTS_TTL_HOURS`：结果文件保留时长（用于清理）

- **`config.toml`（适合 CLI/本地）**
  - 内容与 `.env` 等价，只是结构化。

## 5. API 设计（远程调用）
### 5.1 约定
- API 版本前缀：`/v1`
- 错误返回：统一 JSON 结构（错误码、错误信息、可选详情）
- 支持输入：文件上传（multipart）与 URL（二选一）

鉴权约定：

- 默认启用 **API Key**。
- 推荐使用 HTTP Header：`Authorization: Bearer <API_KEY>`。

### 5.2 ASR
- `POST /v1/asr/transcribe`
  - 输入：音频文件或 `audio_url`，可选参数：`model_name`、`language`、`batch_size`、`timestamps`、`flash`、`diarization`、`hf_token`、speaker 参数等
  - 输出：
    - `text`：全文
    - `chunks`：带时间戳分段（或 word 级）
    - `speaker_segments`：开启 diarization 时返回（包含说话人标签与时间范围）
    - `meta`：模型、设备、耗时等

- `POST /v1/asr/translate`
  - 与上类似，`task=translate`

### 5.3 TTS
- `POST /v1/tts/synthesize`
  - 输入：`text` + 可选 `voice_prompt`（参考音频），以及采样/控制参数（如 temperature、top_p 等）
  - 输出：
    - 同步小文件可直接返回二进制（`audio/wav` 或 `audio/ogg` 或 `video/mp4`）
    - 默认推荐走“结果文件下载”模式（见 5.5）

### 5.4 运维接口
- `GET /healthz`：进程存活
- `GET /readyz`：模型是否可用/是否已加载
- `POST /v1/admin/warmup`：预下载/预热（可选）

### 5.5 结果文件交付与下载（建议）
你的使用场景是：服务部署在带显卡的主机上，远程调用方需要把结果下载到本地。因此建议 API 支持“生成结果文件 + 下载”的工作流。

推荐两种模式：

- **同步模式（默认，适合小文件实时转换 + 同步下载）**
  - 直接返回：
    - ASR：返回 JSON（`text/chunks/speaker_segments/meta`）
    - TTS：返回音频二进制（`audio/wav` 或 `audio/ogg` 或 `video/mp4`）
  - 适用：音频较短、请求方希望立刻拿到结果的场景。
  - 备注：同步模式通常不落盘（可选），不依赖 TTL。

- **异步任务 + 下载（适合大文件/耗时任务）**
  - `POST /v1/asr/transcribe` / `POST /v1/tts/synthesize` 支持参数 `mode=async`。
  - 返回：`task_id` + `status_url`（当任务完成后 `results[]` 提供下载链接）。
  - 新增接口：
    - `GET /v1/tasks/{task_id}`：查询状态、错误信息、产物列表
    - `GET /v1/results/{result_id}`：下载产物（如 `output.json`、`output.wav`、`output.ogg`、`output.mp4`）
  - 服务端：把产物保存在 `LIBER_RESULTS_DIR`，并按 `LIBER_RESULTS_TTL_HOURS` 做定期清理。

模式选择建议（可写入服务端策略）：

- 默认 `mode=sync`
- 当满足以下任一条件时建议走 `mode=async`：
  - 音频时长超过阈值（例如 60s/120s，可配置）
  - 请求方显式指定 `mode=async`
  - 预计输出较大（例如生成 mp4）且调用方网络不稳定

## 6. CLI 设计（本地快速调用）
建议命令统一为一个入口：`liber-speech`。

- `liber-speech asr transcribe --file <path|url> [--model ...] [--flash] [--timestamps chunk|word] [--out output.json]`
- `liber-speech asr translate --file <...>`
- `liber-speech tts synthesize --text "..." [--voice-prompt ref.wav] --out out.wav`
- `liber-speech tts synthesize --text "..." [--voice-prompt ref.wav] --format wav|mp4|ogg_opus --out out.ogg`
- `liber-speech serve --host 0.0.0.0 --port 8000`
- `liber-speech warmup`：下载模型/跑一次短样例
- `liber-speech doctor`：诊断 torch/cuda/ffmpeg/缓存目录可写性/可选依赖等

## 7. 交互式安装与配置（uv）
### 7.1 目标
- 一条脚本在 Windows/Linux 上完成：创建 uv 环境、安装依赖、生成配置、可选预热（Windows 优先提供 `setup.ps1`，Linux/macOS 提供 `setup.sh`）。
- 将复杂选项变为交互式选择，减少手工编辑。

### 7.2 交互流程（拟定）
- 选择安装档位：
  - `CPU`
  - `CUDA(NVIDIA)`
- 选择启用组件：
  - ASR
  - TTS
  - diarization（提示需要 HF Token）
  - flash-attn（高级，提示兼容性与风险）
- 选择默认模型：
  - ASR：默认 `openai/whisper-large-v3`（或按你的偏好）
-  - TTS：默认 `multilingual`（可配置切换）
- 选择 TTS 输出格式：`wav` / `mp4` / `ogg(opus)`（用于 Telegram Voice Note）
- 配置 API 鉴权：生成/录入 API Key
- 生成 `.env`/`config.toml`
- 可选：warmup（下载模型并做一次推理自检）

### 7.3 依赖策略（拟定）
- 最小可用依赖：ASR(TF pipeline) + TTS(chatterbox-tts) + FastAPI + Typer。
- diarization、flash-attn 作为可选 extras。

## 9. 风险与对策
- **CUDA/Torch 兼容（Windows）**：提供 `doctor` 命令与脚本提示；必要时引导安装正确的 torch CUDA wheel。
- **模型下载耗时/体积大**：提供 warmup；支持自定义缓存目录。
- **flash-attn 安装复杂**：作为可选项，脚本默认关闭；启用时给出明确风险提示。
- **diarization 需要 HF Token**：仅在开启时提示配置 token。

## 10. 已确认的关键决策与推荐参数
### 10.1 远程 API 鉴权（已确认）
- 固定采用：`Authorization: Bearer <API_KEY>`。

### 10.2 TTS 的 MP4 输出（已确认）
- 无特殊要求，按常规音频封装即可。
- 建议实现：**audio-only MP4**，默认编码 **AAC-LC**（兼容性最佳）。

### 10.3 Telegram Voice Note（`ogg/opus`）最佳实践（推荐）
Telegram 的语音消息（Voice Note）对 **OGG/Opus** 兼容性最好。建议统一采用以下参数生成：

- **封装**：`OGG`
- **编码**：`Opus`
- **声道**：`mono`（1 声道，语音最优且体积更小）
- **采样率**：`48kHz`（Opus 原生采样率，避免平台侧再重采样）
- **码率（VBR）**：建议 `24kbps` ~ `32kbps`
  - 语音场景下 `24kbps` 通常足够清晰
  - 对音质更敏感可用 `32kbps`
- **Opus 应用模式**：`voip`
- **响度**：建议输出前做一次响度归一化（例如目标 `-16 LUFS` 左右，用于提升语音一致性）

实现建议（落地层面）：

- 服务端统一先生成 `wav`（模型原始输出），再用 `ffmpeg` 转码生成 `ogg/opus` 与 `mp4`。
- `ogg/opus` 文件扩展名使用 `.ogg`。

### 10.4 结果文件保存与下载（仍需你给一个数值）
- 已确认：默认走“同步模式（sync）”。
- 当使用 `mode=async` 时，才会在服务端落盘保存产物并提供下载链接。

TTL 含义说明：

- **TTL（Time To Live）**：结果文件（转录 JSON、wav/ogg/mp4 等）在服务端磁盘上“最多保留多久”。
- TTL 仅对 **异步任务 + 下载** 的落盘产物生效；同步模式通常不落盘，因此不受 TTL 影响。
- 作用：
  - 避免服务器磁盘长期堆积被占满
  - 给调用方留出“下载窗口期”（在 TTL 时间内可下载；超过 TTL 可能被清理而无法下载）

推荐默认值（可按你的业务调整）：

- **默认 24 小时**：磁盘压力最小，适合调用方能快速拉取结果的场景
- **72 小时**：兼顾稳定性与磁盘占用，适合偶发重试/跨时区下载
- **7 天**：最省心但磁盘占用更高，适合低频任务或强审计留存

配置方式：

- 使用 `LIBER_RESULTS_TTL_HOURS` 设置小时数，例如：
  - 24 小时：`LIBER_RESULTS_TTL_HOURS=24`
  - 72 小时：`LIBER_RESULTS_TTL_HOURS=72`
  - 7 天：`LIBER_RESULTS_TTL_HOURS=168`

### 10.5 异步任务模式：如何确认“转换任务”和“生成结果”的对应关系
异步模式的核心是：**提交任务返回 `task_id`，所有产物都挂在该 `task_id` 下**。

建议约定：

- 提交接口（示例）：`POST /v1/asr/transcribe?mode=async` 或 `POST /v1/tts/synthesize?mode=async`
- 返回字段：
  - `task_id`
  - `status_url`：`/v1/tasks/{task_id}`

任务状态查询：

- `GET /v1/tasks/{task_id}` 返回任务状态与产物清单。
- 建议状态机（`status`）：
  - `queued`：已入队等待
  - `running`：处理中
  - `succeeded`：成功（产物可下载）
  - `failed`：失败（包含错误信息）
  - `expired`：已过期/被清理（TTL 到期）

建议返回结构（示例字段，具体实现可调整）：

- `task_id`
- `status`
- `created_at` / `started_at` / `finished_at`
- `progress`：0~100（可选）
- `error`：失败时的错误对象（可选）
- `results`：产物数组，每个产物包含
  - `result_id`
  - `type`：`asr_json` / `tts_wav` / `tts_ogg_opus` / `tts_mp4`
  - `filename`
  - `content_type`
  - `size_bytes`
  - `download_url`：`/v1/results/{result_id}`
  - `expires_at`：该产物的过期时间（与 TTL 对齐）

下载产物：

- `GET /v1/results/{result_id}`
  - 返回二进制文件（或 `Content-Disposition: attachment` 让客户端保存）

调用方（例如远程 openclaw）确认方式：

- 记录 `task_id`
- 轮询 `GET /v1/tasks/{task_id}` 直到 `status=succeeded`
- 从 `results[]` 中选择需要的格式并按 `download_url` 下载到本地
